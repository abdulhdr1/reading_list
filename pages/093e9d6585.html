A Tale of Two Agent Builders

13/10/2025 - 01:00
Visit original ‚Üó

This article compares two fundamentally different approaches to building AI agent configuration interfaces, released in the same week: OpenAI's canvas-based Agent Builder and Intercom/Fin's document-based Procedures. OpenAI's Agent Builder uses the classic 'nodes and noodles' visual workflow pattern ‚Äî a canvas where you drag, connect, and configure logic gates in an if-then-else flow. It's a well-established pattern with deep roots (Quartz Composer, Max/MSP, Scratch, Zapier, n8n) but inherently deterministic. The initial reaction was mixed: it felt like squeezing massively capable agentic systems through narrow logic gates. As workflows grow complex, they become spaghetti. LLMs work well with language, not flowcharts. Impressively (or tellingly), it was built in just six weeks. Fin's Procedures takes the opposite approach: a document-based editor inspired by Jupyter Notebooks and Notion. You start by describing what you want in plain English (probabilistic), then progressively add deterministic elements inline ‚Äî branching logic, data connectors, code blocks ‚Äî only when needed. It echoes Stephen Wolfram's aspiration of 'programming in which one is as directly as possible expressing computational thinking.' The result is a blend of agentic and specific, with progressive complexity. The deeper insight is about the central UI design challenge of the AI era: how do we combine deterministic interfaces (predictable, if-then-else) with probabilistic ones (fuzzy, AI-driven)? The author believes the primary user interfaces of the future will be the ones that most gracefully blend both. OpenAI is building generic multi-agent infrastructure; Fin is building domain-specific customer support tooling ‚Äî different goals explaining different design choices, but both wrestling with the same fundamental tension.
